# テストの作成方針
基本的に以下2種類に分類する。

- 1関数毎の単体テスト
    - 内部の分岐を網羅するようテストする
        - 挙動が非`static`な関数の分岐に依存するような場合、`mock`を利用する
        - `static`な関数の場合も、制御が難しければ`mock`を利用する
    - 外側のクラスは`~Test`と命名する
- エンドポイントを通しで実行する結合テスト
    - テストは`~Routes`毎に作成するため、テストクラスは`~RoutesIntegrationTest`となる

作成方針は、単体テストが基本的に必須、結合テストは必要に応じて作成とする。

## クラス定義とリフレクションの利用について
基本的にテストのためのリフレクション利用は禁止する。  
このため、テストが必要な関数は何らかの形で公開することとする。

ただし、テストが`enum`や`sealed class`の定義を網羅しているか担保するためのチェックに関しては、リフレクションを利用してもよい。

## 単体テストの組み方について
基本的に1関数1`NestedClass`、`NestedClass`内で分岐を網羅出来るようでテストを作成する。  
このネストには`JUnit`の`@Nested`アノテーションを使用する。  
ただし、1関数で全ケースを網羅出来る場合、`NestedClass`は定義せず1関数にまとめてもよい。

変数定義は、なるべく外側に共通で定義し、`NestedClass`でしか使わない変数は`NestedClass`側に定義する。

テストクラス名は`${先頭を大文字とした関数名}Test`を基本とする。  
コンストラクタのテストについては、`InitTest`と命名する。

1つ1つのケースは、検証内容をテスト名とする。

### コンストラクタで`mock`を要求するテストについて
テスト対象の変数名は`sut`とする。  
内部の関数分割状況によっては`spy`し、後述する方針に沿って内部関数を`mock`・呼び出しチェックする。

### DBアクセスするテストの組み方について
DBアクセスするテストは`mock`化をしない。  
また、ローカルのデータは破壊してよいものと扱う。

テストデータは、投入用`SQL`ファイルを別で定義し、実行毎に投入する。

これは単体テストとして扱う。

対象毎に、`~RepositoryImplTest`や`~QueryServiceImplTest`と命名する。  
また、テスト対象となる`Impl`の変数名は`sut`とする。  
内部の関数分割状況によっては`spy`し、後述する方針に沿って内部関数を`mock`・呼び出しチェックする。

# mock化について
`mock`化には`mockk`を利用する。

`every`や`justRun`等による関数呼び出しの`mock`時、`capture`等が不要なら、引数は`any`を指定する。

## static mockについて
`mockkObject`や`mockkStatic`は、特殊なケースを除き基本的に`BeforeEach`で行う。  
また、それを行った場合`AfterEach`でモック化を解除する。

`mock`化した関数をテスト内で直接呼び出すことは禁止。  
戻り値が必要な場合、一時変数に格納して使いまわす必要が有る。

# 比較について
## 期待値の比較について
基本的に`assertEquals`一発で対象インスタンス全体を比較する。  
ただし、コンストラクタが`private`な`class`の生成結果を比較する場合、変更後のテスト追加漏れを避けるため、
`toString`の結果と期待値文字列の比較で実装を行う。

## 呼び出し回数の比較について
非`static`な`mock`関数は、呼び出し回数チェックを必須とする。

呼び出し回数として1回以上が期待される場合、`verify`で確認する。  
この時、基本的に引数は厳密な一致を確認する。  
また、呼び出し回数も明示的に指定する。

0回が期待される場合、回数が0、引数は`any`で比較する。
